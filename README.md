# 数字员工 · 阿桃｜小红书运营专员（MVP v1）

每天交付 **3 套能直接发布的小红书笔记**：成品封面图（1242×1660 PNG）+ 打分筛选后的标题 Top3 + 正文 + 标签 + 发布建议。

不是聊天机器人——是一条「选题引擎 → 文案引擎 → 质检筛选 → 封面工厂 → 交付打包」的生产流水线，外加工牌、入职培训、工作日志的完整员工体验。

---

## 一、部署（Netlify，3 步）

1. 把本项目推到 GitHub 仓库（或直接拖 zip 到 Netlify）
2. Netlify → New site → 选仓库，构建命令 `npm run build`（netlify.toml 已配好，自动识别 Next.js）
3. Site settings → Environment variables，配置：

| 变量 | 必填 | 说明 |
|---|---|---|
| `LLM_API_BASE` | ✅ | OpenAI 兼容网关地址，到 `/v1` 为止（零克云网关） |
| `LLM_API_KEY` | ✅ | 网关 API Key |
| `LLM_MODEL` | ✅ | 生产用模型（写作主力，建议长文能力强的） |
| `LLM_MODEL_JUDGE` | 可选 | 质检环模型，建议用便宜快的；留空则与 LLM_MODEL 相同 |

本地开发：`cp .env.example .env.local` 填好变量 → `npm install` → `npm run dev`

生产流程：选题（1 次调用）→ 3 篇并行生产（每篇：10 标题+正文+封面文案 1 次调用）→ 质检评分（每篇 1 次，可独立模型）→ 代码侧硬规则合并打分（≤20字 / 违禁词 / 钩子位 / 具体度）→ 10 进 3 →正文违禁词复查 → 交付。单篇失败不拖垮整批。

## 三、产品设计要点

- **筛选过程可见**：每张交付卡都能展开"被淘汰的 7 条及原因"——会自我淘汰废稿，是员工和聊天框的分水岭
- **封面是成品不是建议**：HTML 模板排版 + 原尺寸导出，AI 不直接画中文字（画不好）
- **违禁词分级**：block 级（医疗功效/站外引流/绝对化承诺）硬标红，warn 级提示复查，规则在 `bannedWords.js` 随平台政策维护
- **当日交付持久化**：localStorage 按日期存档，刷新不丢；「重新培训」可改入职档案

## 四、已知边界（v1 刻意不做的）

- **热点为半自动投喂**：工作台顶部贴热点/对标即可参与选题；自动拉取需接搜索 API 或数据服务（蝉妈妈/新红等），留到 v2
- **封面暂无图片底版式**：对比图/产品图/九宫格等 12 个版式需接生图模型（走零克云生图 API）后扩展
- **定时日更未开**：当前手动开工；自动日更需要后端存储（不能依赖 localStorage），方案：Netlify Scheduled Functions + KV/数据库，v2 一并做
- **单用户单账号**：无登录体系，档案存浏览器本地

## 五、路线图

v1.1 数据复盘闭环（贴笔记数据截图 → 阿桃复盘调策略）→ v1.2 生图背景层 + 图片版式 → v1.3 定时日更 + 交付通知 → v2 抖音运营专员（资产库已备好：Hook24式/剧本5模板/质检10条）→ v3 花名册扩编与多员工协作
